Generated from Webflow CMS exports by `scripts/transform-cms.py`.

## Key Naming Convention

All JSON files use **flat key-value pairs** wherever possible. This format is natively compatible with [Tolgee](https://tolgee.io) for Phase 2 translation management.

### Top-level keys (every content item)

| Key | Type | Locale-specific? | Description |
|---|---|---|---|
| `slug` | string | No ŌĆö shared | URL slug (e.g. `google-business-profile`) |
| `locale` | string | No ŌĆö this file's locale | `en`, `de`, `fr`, `es` |
| `collection_type` | string | No ŌĆö shared | Collection folder name |
| `updated_at` | string | No ŌĆö shared | ISO 8601 date from Webflow |

### SEO block (nested object ŌĆö required on every page/collection item)

```json
{
  "seo": {
    "title": "Page title for <title> tag",
    "description": "Meta description (150ŌĆō160 chars)",
    "canonical": "https://www.synup.com/[locale-prefix]/[collection]/[slug]",
    "ogImage": "https://cdn.prod.website-files.com/..."
  }
}
```

SEO fields are **locale-specific** ŌĆö each locale file has its own title and description.
The `canonical` URL is set by the transform script and reflects the locale URL structure.

### Content fields (locale-specific)

Naming follows `section_field` or plain `field` for top-level copy:

| Pattern | Example keys | Notes |
|---|---|---|
| `name` | `name` | Display name of item |
| `title` | `title` | Page/item title |
| `headline` | `headline` | Hero headline |
| `subheadline` | `subheadline` | Hero subheadline |
| `description` | `description` | Short description |
| `body` | `body` | Long-form content (sanitized HTML) |
| `cta_label` | `cta_label` | Call-to-action button text |
| `cta_href` | `cta_href` | CTA destination URL |
| `excerpt` | `excerpt` | Short teaser text |

### Shared (non-locale-specific) fields

These fields should be **identical across all 4 locale files** for the same item:

- `slug` ŌĆö URL slug
- Any field matching: `*_url`, `*_id`, `*_ref`, `*_date`, `*_rating`, `*_score`, `*_count`, `*_order`
- Image/logo URLs (`*_image`, `*_logo`, `*_photo`, `*_thumbnail`)
- External links

### Rich text fields

Rich text fields contain **sanitized HTML**. Allowed tags: `p`, `br`, `strong`, `b`, `em`, `i`, `u`, `ul`, `ol`, `li`, `h1`ŌĆō`h6`, `a`, `blockquote`, `span`.

All `<script>`, `<style>`, event handlers (`on*`), and `javascript:` hrefs are stripped at transform time.

In React, render rich text via a sanitization wrapper ŌĆö **never use `dangerouslySetInnerHTML` without `DOMPurify`**.

## Placeholder Values

The transform script uses two special placeholder strings for human review:

### `__MISSING__`

A field that has **no value at all** in the Webflow export for this locale. This is a genuine content gap.

```bash
# Find all missing fields
grep -r "__MISSING__" content/
```

Action: Fill in the correct translation manually and re-commit.

### `__POSSIBLE_UNTRANSLATED__`

A field that appears to **still be in English** in a non-EN locale file. The original English text is preserved verbatim after the prefix.

```json
{
  "headline": "__POSSIBLE_UNTRANSLATED__Get more customers with Synup"
}
```

```bash
# Find all possibly untranslated fields
grep -r "__POSSIBLE_UNTRANSLATED__" content/
```

Action: Verify whether the text is actually translated (sometimes EN terms are correct in other locales), then either keep it or supply the correct translation.

## How to Add a New Content Type

1. **Add the collection to the transform script**
   In `scripts/transform-cms.py`, add an entry to `COLLECTION_NAME_MAP`:
   ```python
   "My New Collection - ": "my-new-collection",
   ```

2. **Add a TypeScript interface**
   In `lib/content-types.ts`, define a new interface extending `PageContentBase`:
   ```typescript
   export interface MyNewCollection extends PageContentBase {
     collectionType: "my-new-collection";
     title: string;
     // ... other fields
     seo: SeoMeta;
   }
   ```
   Add it to the `COLLECTION_TYPES` constant and `AnyCollectionItem` union.

3. **Export the Webflow collection as xlsx**
   Place the xlsx in each locale's subfolder under `/exports/csv-exports-cms/`.

4. **Run the transform script**
   ```bash
   cd synup-nextjs
   python3 scripts/transform-cms.py
   ```
   JSON files will be written to `/content/[locale]/my-new-collection/`.

5. **Update Phase 2 Tolgee keys** (post-launch only)
   Export the new JSON files to Tolgee using their CLI:
   ```bash
   tolgee push --path content/en/my-new-collection
   ```

## Tolgee Compatibility (Phase 2)

This content layer is designed for zero-rework Tolgee migration:

- **Flat JSON format**: Tolgee natively imports flat key-value JSON files
- **Locale folders**: Match Tolgee's expected structure (`/en/`, `/de/`, `/fr/`, `/es/`)
- **Key naming**: `section.field` pattern ŌĆö compatible with Tolgee key management
- **No nested translation strings**: All copy is at the root or one level deep in `seo`

To import existing content into Tolgee (Phase 2 setup):
```bash
# For each locale and collection:
tolgee push --path content/en/ --language en
tolgee push --path content/de/ --language de
tolgee push --path content/fr/ --language fr
tolgee push --path content/es/ --language es
```

Tolgee will detect new English keys and queue them for DeepL auto-translation.

## Regenerating Content

If source xlsx files change, re-run the transform script:

```bash
cd synup-nextjs
pip3 install openpyxl   # first time only
python3 scripts/transform-cms.py
```

The script is **idempotent** ŌĆö it overwrites existing files. Check the generated `content/transform-report.json` for any new missing or flagged fields.

## Security Notes

- JSON files contain no executable code
- Rich text fields are sanitized at transform time (see `sanitize_html()` in transform script)
- Slug validation prevents path traversal
- Content loader (`lib/content.ts`) validates slugs before constructing file paths
- Never pass raw JSON field values to `dangerouslySetInnerHTML` without DOMPurify
